Yes, **Firefox can support H.265 through WebCodecs**, but this is the messy part:

**Normal Firefox HEVC playback is more mature than Firefox HEVC WebCodecs support.** The `<video>` / MSE path is the safer bet today.

### The key distinction

There are two different decode paths:

| Use case         | API                                     | Firefox HEVC status                                                          |
| ---------------- | --------------------------------------- | ---------------------------------------------------------------------------- |
| Normal playback  | `<video src="...">`, MSE, media element | Supported on modern Firefox when platform codecs/hardware are available      |
| Low-level decode | `VideoDecoder` from WebCodecs           | Exists/has been landing, but more conditional and historically Nightly-first |

MDN’s Firefox 137 notes say HEVC is hardware-enabled on Android, hardware/software-enabled on Linux, and that this adds to existing Windows/macOS support. That is about Firefox media playback support generally. ([MDN Web Docs][1])

WebCodecs is different: it gives JS low-level access to `VideoDecoder` / `VideoEncoder`, turning `EncodedVideoChunk`s into `VideoFrame`s directly, often with hardware acceleration. ([MDN Web Docs][2])

### Do you have to use a video element?

**No, not if WebCodecs H.265 works on that Firefox/platform.**

With WebCodecs you can do:

```js
const support = await VideoDecoder.isConfigSupported({
  codec: 'hvc1.1.6.L93.B0', // example HEVC codec string
});

console.log(support.supported);
```

Then decode chunks manually:

```js
const decoder = new VideoDecoder({
  output: frame => {
    // draw to canvas, WebGL, WebGPU, etc.
    frame.close();
  },
  error: err => console.error(err),
});

decoder.configure({
  codec: 'hvc1.1.6.L93.B0',
});

decoder.decode(new EncodedVideoChunk({
  type: 'key',
  timestamp: 0,
  data: encodedHevcBytes,
}));
```

But in practice, for Firefox, **you should assume `<video>` / MSE is more reliable than WebCodecs for HEVC**, especially if you need this to work across Windows, macOS, Linux, and Android.

### How Firefox supports HEVC

Firefox does **not** simply bundle a universal “Mozilla HEVC decoder” everywhere. It mostly relies on platform/media decode stacks:

```text
H.265 / HEVC media
        ↓
Firefox media pipeline
        ↓
Platform decoder
Windows: Media Foundation / HEVC codec support
macOS: Apple VideoToolbox
Linux: FFmpeg / VA-API path depending on distro/build
Android: platform hardware decoder
        ↓
GPU / hardware decode when available
```

Mozilla specifically said the plan for H.265 WebCodecs was to support pass-through/platform-backed decoding, and that H.265 support would only happen where the platform supports it, such as hardware acceleration. ([bugzilla.mozilla.org][3])

### Current practical answer

For your use case:

* **For simple playback:** use `<video>` or MSE.
* **For low-level frame access:** try WebCodecs, but feature-detect it.
* **For broad compatibility:** provide an H.264 or AV1 fallback.
* **For Firefox specifically:** do not assume HEVC WebCodecs works just because HEVC `<video>` playback works.

Firefox Nightly release notes show H.265 WebCodecs decode being enabled first in Nightly macOS builds starting Firefox 137 and Nightly Linux builds starting Firefox 138, which is a clue that this path has been more experimental/staged than normal media playback. ([Firefox][4])

### Best runtime checks

For `<video>` playback:

```js
const video = document.createElement('video');

console.log(video.canPlayType(
  'video/mp4; codecs="hvc1.1.6.L93.B0"'
));
```

For MSE:

```js
console.log(MediaSource.isTypeSupported(
  'video/mp4; codecs="hvc1.1.6.L93.B0"'
));
```

For WebCodecs:

```js
const supported =
  'VideoDecoder' in window &&
  (await VideoDecoder.isConfigSupported({
    codec: 'hvc1.1.6.L93.B0',
  })).supported;

console.log(supported);
```

The rule of thumb:

```text
<video> HEVC works
  does NOT guarantee
WebCodecs HEVC works
```

So yes, **Firefox can support H.265 via WebCodecs**, but for production today I would treat it as **detect-and-use**, not **assume-and-rely-on**.
