That new error is from your **`models.json` shape**, not the original Copilot API error.

Two likely issues:

1. You have a typo:
   `github-coilot` → should be `github-copilot`

2. Pi does **not** accept a provider entry that only contains top-level `compat`. The validator says a provider must include one of `baseUrl`, `headers`, `compact`, `modelOverrides`, or `models`. Pi’s docs show `compat` is supported inside `modelOverrides`, and `modelOverrides` is specifically for customizing built-in provider models without replacing the full model list. Unknown model IDs are ignored, so the ID has to match the model you are actually using. ([Pi][1])

Use this format instead:

```json
{
  "providers": {
    "github-copilot": {
      "modelOverrides": {
        "claude-haiku-4-5": {
          "compat": {
            "supportsEagerToolInputStreaming": false
          }
        }
      }
    }
  }
}
```

Replace `claude-haiku-4-5` with the exact Copilot model ID Pi shows for the model you selected.

The reason this fixes the original error is that Pi normally sends `tools[].eager_input_streaming`; Pi’s docs say if a proxy or Anthropic-compatible backend rejects that field, setting `supportsEagerToolInputStreaming: false` makes Pi omit it and use the legacy fine-grained tool streaming header instead. ([Pi][1])

```bash
cat ~/.pi/agent/models.json | jq .
```

Then list models so you can get the exact model ID:

```bash
pi --list-models | grep -i copilot
```

Then edit:

```bash
nano ~/.pi/agent/models.json
```

Final file should look like:

```json
{
  "providers": {
    "github-copilot": {
      "modelOverrides": {
        "EXACT_MODEL_ID_HERE": {
          "compat": {
            "supportsEagerToolInputStreaming": false
          }
        }
      }
    }
  }
}
```

Most important part: **do not put `compat` directly under `github-copilot` by itself**. Put it under:

```text
providers → github-copilot → modelOverrides → exact-model-id → compat
```
