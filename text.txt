The biggest thing to know is that **delegation is not mostly controlled by your chat prompt**. Copilot’s docs recommend defining **when to use subagents inside the custom agent instructions**, not manually re-describing delegation every time. That prompt is designed around how Copilot actually works today:

* custom agents can specify a **single model or a prioritized list of models**; if `model` is omitted, the current model picker is used instead. ([Visual Studio Code][1])
* worker-only agents should usually use `user-invocable: false`. Agents that should never be auto-used as subagents can use `disable-model-invocation: true`. ([Visual Studio Code][1])
* reliable delegation comes from **explicit `agents` allow-lists** on the coordinator, not from letting Copilot choose from every vaguely similar agent. The docs specifically warn that otherwise it may select an unintended agent. ([Visual Studio Code][1])
* if you want prompt files to trigger subagents, they need `agent` or `runSubagent` in frontmatter. ([Visual Studio Code][2])

For “how do I make sure stuff is delegated correctly,” these are the levers that matter most:

First, make the **top-level orchestrator explicit**. The VS Code docs show a coordinator/worker pattern where one agent owns the workflow and delegates to narrowly scoped workers. That is much more reliable than hoping any user-facing agent will spontaneously pick the right helper. ([Visual Studio Code][2])

Second, keep worker roles **sharp and non-overlapping**. If two agents both sound like “reviewer” or both sound like “implementer,” the main agent can choose the wrong one. The docs call this out directly. ([Visual Studio Code][2])

Third, make delegation rules part of the **agent body**, not just your one-off chat prompt. VS Code explicitly recommends defining when to use subagents in the custom agent’s instructions for consistent behavior. ([Visual Studio Code][2])

Fourth, use **handoffs for user-guided phase changes**, not as your only orchestration mechanism. Handoffs are great in VS Code for plan → implement → review, but GitHub.com’s coding agent ignores `handoffs`, so the instructions and subagent design still need to stand on their own. ([Visual Studio Code][1])

One honest caveat: **Copilot cannot reliably know which exact models your org has enabled just from repo files**. GitHub’s docs show you can set `model:` in the `.agent.md` file and use autocomplete in the agent editor, and VS Code now supports **multiple-model fallback arrays**. But exact model names still need to match what your installation exposes. So the safest flow is: let Copilot wire the architecture and intended routing, then you do one quick pass in the agent editor to normalize any model names that do not match your picker/autocomplete. ([GitHub Docs][3])

A good default routing policy is:

* **Planner / Architecture Checker / Reviewer / Platform Investigator**: strongest reasoning-oriented model you have.
* **Bugfixer / UI Implementer / Test Author**: faster coding-oriented model, with a stronger fallback if available.
* **Coordinator**: strong general-purpose model, or no explicit `model:` if you want the current picker to control it. GitHub’s model comparison docs describe this general split between deep reasoning/debugging models and faster/simple-task models. ([GitHub Docs][4])
