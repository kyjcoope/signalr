You *can* use Qwen, but I would **not use it through GBrain’s `ollama:` provider** right now. The cleanest way is:

```text
Qwen3 Embedding GGUF
→ llama.cpp llama-server
→ GBrain llama-server provider
```

Reason: GBrain’s current Ollama recipe is built around known Ollama embedding models like `nomic-embed-text` 768d, `mxbai-embed-large` 1024d, and `all-minilm` 384d. Its docs list Ollama as defaulting to 768d, while `llama-server` is the local provider meant for user-set dimensions. ([GitHub][1])

Qwen3 Embedding is real and good for this use case: it is specifically designed for text retrieval, code retrieval, classification, clustering, and multilingual search. Qwen’s 0.6B model outputs up to **1024 dimensions**, 4B outputs up to **2560**, and 8B outputs up to **4096**. ([Hugging Face][2])

## The short recommendation

Use:

```text
Qwen3-Embedding-0.6B
1024 dimensions
llama-server provider
```

Do **not** use 4B first. 4B is 2560 dimensions, and GBrain’s docs note pgvector HNSW supports up to 2000 dimensions; above that it falls back to exact vector scans, which is correct but slower. ([GitHub][1])

# Option A — Try Qwen through Ollama directly

This is the simple version, but it may hit the same 768/1024 mismatch you already saw.

```bash
ollama pull qwen3-embedding:0.6b

curl http://localhost:11434/api/embed \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen3-embedding:0.6b","input":"test"}' \
  | jq '.embeddings[0] | length'
```

Expected:

```text
1024
```

Then try:

```bash
gbrain init --force --pglite \
  --embedding-model ollama:qwen3-embedding:0.6b \
  --embedding-dimensions 1024

gbrain providers test --touchpoint embedding
gbrain doctor
```

If that works, great.

If it says something like:

```text
Provider "ollama" model "qwen3-embedding:0.6b" does not support custom dimensions
```

or:

```text
model returned 1024 but schema expects 768
```

then stop using the `ollama:` provider for Qwen and use Option B.

Ollama currently supports embedding models and the `/api/embed` endpoint, but custom dimension handling for Qwen-style Matryoshka embeddings has been a known pain point; there is an Ollama issue requesting a `dimensions` parameter for embedding models like Qwen3 because Qwen3 defaults can exceed common pgvector limits. ([GitHub][3])

# Option B — Recommended: Qwen through `llama-server`

## 1. Install llama.cpp on macOS

```bash
brew install llama.cpp
```

Qwen’s own llama.cpp docs say macOS/Linux users can install `llama-cli` and `llama-server` with Homebrew using `brew install llama.cpp`. They also note llama.cpp supports Apple Silicon via Metal/Accelerate. ([Qwen][4])

Verify:

```bash
which llama-server
llama-server --version
```

## 2. Download Qwen3 Embedding 0.6B GGUF

Install Hugging Face CLI:

```bash
python3 -m pip install --user huggingface_hub
```

Make sure Python user binaries are on PATH:

```bash
export PATH="$HOME/Library/Python/3.*/bin:$PATH"
```

The wildcard may not expand in every shell, so if `huggingface-cli` is not found, run:

```bash
python3 -m site --user-base
```

Then add the printed path plus `/bin` to PATH.

Download the model:

```bash
mkdir -p ~/models/qwen3-embedding-0.6b
cd ~/models/qwen3-embedding-0.6b

huggingface-cli download Qwen/Qwen3-Embedding-0.6B-GGUF \
  --local-dir .
```

If that downloads multiple quantizations and you want a specific one, list files:

```bash
find . -name "*.gguf" -maxdepth 2
```

Pick a Q8 or F16 file if available. The 0.6B model is small enough that Q8 is fine on most Macs.

## 3. Start `llama-server` for embeddings

Find the GGUF:

```bash
MODEL_PATH="$(find ~/models/qwen3-embedding-0.6b -name '*.gguf' | head -n 1)"
echo "$MODEL_PATH"
```

Run:

```bash
llama-server \
  --model "$MODEL_PATH" \
  --embedding \
  --pooling last \
  --host 127.0.0.1 \
  --port 8080
```

Leave that terminal open.

Qwen discussion examples for llama.cpp use `llama-server --model /path/Qwen3-Embedding-0.6B-*.gguf --embedding --pooling last --port 8080`, then call `/v1/embeddings`. ([Hugging Face][5])

## 4. Test Qwen embeddings from `llama-server`

In a second terminal:

```bash
curl http://localhost:8080/v1/embeddings \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3-embedding-0.6b",
    "input": "GBrain stores project decisions, repo memory, and workflow notes."
  }' | jq '.data[0].embedding | length'
```

Expected:

```text
1024
```

## 5. Initialize GBrain using `llama-server`

For a **fresh/empty brain**:

```bash
gbrain init --force --pglite \
  --embedding-model llama-server:qwen3-embedding-0.6b \
  --embedding-dimensions 1024
```

Then:

```bash
gbrain providers test --touchpoint embedding
gbrain doctor
```

GBrain’s embedding-provider docs say the `llama-server` provider is local, user-set dimensionality, and should be used by launching `llama-server` with `--model <gguf-path> --embeddings`, then initializing GBrain with `--embedding-model llama-server:<your-id> --embedding-dimensions <N>`. ([GitHub][1])

## 6. Import and embed your brain

```bash
gbrain import ~/ai-brain
gbrain embed --stale
gbrain query "What is this brain for?"
```

# If you already have an existing GBrain brain

Check whether it has content:

```bash
gbrain doctor
gbrain stats 2>/dev/null || true
```

If it is basically empty, easiest path:

```bash
gbrain init --force --pglite \
  --embedding-model llama-server:qwen3-embedding-0.6b \
  --embedding-dimensions 1024

gbrain import ~/ai-brain
gbrain embed --stale
```

If it already has embedded content you care about, use GBrain’s migration/reindex path:

```bash
gbrain retrieval-upgrade \
  --to llama-server:qwen3-embedding-0.6b \
  --reindex
```

GBrain’s docs say empty brains can be reinitialized at the right dimension, while non-empty brains should use `gbrain retrieval-upgrade --to <provider>:<model> --reindex`. ([GitHub][1])

If Claude Code/Pi starts GBrain, you need `llama-server` running first.

Update your `~/bin/gbrain-mcp`:

```bash
cat > ~/bin/gbrain-mcp <<'EOF'
#!/usr/bin/env bash

```

Then:

```bash
claude mcp remove gbrain
claude mcp add --scope user gbrain -- "$HOME/bin/gbrain-mcp"
claude mcp list
```

# Make `llama-server` start easily

Create a launcher:

```bash
mkdir -p ~/bin

cat > ~/bin/qwen-embedding-server <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

MODEL_PATH="$(find "$HOME/models/qwen3-embedding-0.6b" -name '*.gguf' | head -n 1)"

if [ -z "$MODEL_PATH" ]; then
  echo "No GGUF model found in ~/models/qwen3-embedding-0.6b" >&2
  exit 1
fi

exec llama-server \
  --model "$MODEL_PATH" \
  --embedding \
  --pooling last \
  --host 127.0.0.1 \
  --port 8080
EOF

chmod +x ~/bin/qwen-embedding-server
```

Run it:

```bash
~/bin/qwen-embedding-server
```

Then in another terminal:

```bash
gbrain providers test --touchpoint embedding
```

# Bottom line

Use Qwen like this:

```text
Best first Qwen setup:
Qwen3-Embedding-0.6B GGUF
via llama-server
1024 dimensions
GBrain provider: llama-server:qwen3-embedding-0.6b
```

Avoid this for now:

```text
GBrain provider: ollama:qwen3-embedding:0.6b
```

because that is where you already saw the dimension mismatch. GBrain’s `llama-server` provider is the intended local escape hatch for user-controlled GGUF embedding models and user-set dimensions.
