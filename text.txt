Yes — here is the full **macOS-local GBrain setup** using **local Ollama embeddings**, with commands for Claude Code and a Pi harness-style setup.

This version assumes everything lives on the Mac:

```text
~/ai-brain       = your Markdown memory repo
~/.gbrain        = GBrain local PGLite database/config
Ollama           = local embedding provider
Claude Code/Pi   = local MCP clients
```

GBrain’s own CLI standalone path is basically: install with Bun, `gbrain init --pglite`, `gbrain doctor`, import Markdown, then query it. The README describes PGLite as the no-server/no-Docker local path. ([GitHub][1])

---

## 0. Install Homebrew if needed

```bash
command -v brew >/dev/null 2>&1 || /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

Load Homebrew into the current shell:

```bash
if [ -x /opt/homebrew/bin/brew ]; then
  eval "$(/opt/homebrew/bin/brew shellenv)"
elif [ -x /usr/local/bin/brew ]; then
  eval "$(/usr/local/bin/brew shellenv)"
fi
```

Persist Homebrew for future shells:

```bash
SHELL_PROFILE="$HOME/.zshrc"

if [ -x /opt/homebrew/bin/brew ]; then
  grep -q 'eval "$(/opt/homebrew/bin/brew shellenv)"' "$SHELL_PROFILE" 2>/dev/null || \
    echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> "$SHELL_PROFILE"
elif [ -x /usr/local/bin/brew ]; then
  grep -q 'eval "$(/usr/local/bin/brew shellenv)"' "$SHELL_PROFILE" 2>/dev/null || \
    echo 'eval "$(/usr/local/bin/brew shellenv)"' >> "$SHELL_PROFILE"
fi

source "$SHELL_PROFILE"
```

Install basic dependencies:

```bash
brew install git jq curl
```

## 1. Install Bun

GBrain is installed via Bun. Bun’s docs say Bun can be installed on macOS/Linux with the official install script and verified with `bun --version` and `bun --revision`. ([Bun][2])

```bash
curl -fsSL https://bun.com/install | bash
```

Add Bun to your current shell:

```bash
export PATH="$HOME/.bun/bin:$PATH"
```

Persist it:

```bash
grep -q 'export PATH="$HOME/.bun/bin:$PATH"' ~/.zshrc 2>/dev/null || \
  echo 'export PATH="$HOME/.bun/bin:$PATH"' >> ~/.zshrc

source ~/.zshrc
```

Verify:

```bash
bun --version
bun --revision
```

## 3. Install Ollama locally on macOS

Ollama’s official macOS docs say the preferred method is installing the `.dmg` app into `/Applications`, and that the app will prompt to create the `/usr/local/bin/ollama` CLI link if needed. They also note macOS Sonoma or newer and Apple Silicon / x86 support. ([Ollama][3])

For a CLI/Homebrew setup:

```bash
brew install ollama
```

Start Ollama:

```bash
brew services start ollama
```

Verify:

```bash
ollama --version
curl http://localhost:11434/api/tags | jq
```

If `brew services` does not start it cleanly, run Ollama manually in a terminal:

```bash
ollama serve
```

Leave that terminal open, then use a second terminal for the rest.

## 4. Pull a local embedding model

For a smooth GBrain setup, start with `nomic-embed-text`. It returns **768-dimensional** embeddings in Ollama, which avoids the Qwen 1024-vs-768 mismatch you hit earlier.

```bash
ollama pull nomic-embed-text
```

Test embeddings:

```bash
curl http://localhost:11434/api/embed \
  -H "Content-Type: application/json" \
  -d '{
    "model": "nomic-embed-text",
    "input": "GBrain stores project decisions, repo memory, and workflow notes."
  }' | jq '.embeddings[0] | length'
```

Expected:

```text
768
```

Ollama’s docs describe embeddings as numeric vectors for semantic search/RAG, and the vector length depends on the model. They also document the `/api/embed` endpoint. ([Ollama][4])

---

## 5. Initialize GBrain with local PGLite + Ollama embeddings

Do **not** pass `--embedding-dimensions` here. Let GBrain use the model’s default.

```bash
gbrain init --pglite --embedding-model ollama:nomic-embed-text
```

Check health:

```bash
gbrain doctor
gbrain providers test --touchpoint embedding
```

If `gbrain providers test` passes, your local embedding setup is good.

## 6. Create your `ai-brain` repo

This is your durable Markdown memory repo. Keep it separate from `~/gbrain`.

```bash
cd ~

mkdir -p ~/ai-brain/{inbox,projects,repos,workflows,decisions,prompts,concepts,sources,people,companies,archive}

cd ~/ai-brain
git init
```

Create starter files:

```bash
cat > README.md <<'EOF'
# AI Brain

Personal/project knowledge base for GBrain.

This repo contains durable project decisions, repo summaries, workflows, prompts, concepts, and source notes.
EOF

cat > RESOLVER.md <<'EOF'
# Resolver

Use this file to decide where new knowledge belongs.

- inbox/ — temporary captures that need sorting
- projects/ — long-running projects
- repos/ — repo-specific architecture summaries and current state
- workflows/ — Claude Code, Codex, Pi, GStack, PR review, CI/CD patterns
- decisions/ — durable decisions and why they were made
- prompts/ — reusable prompts for agents
- concepts/ — explanations and mental models
- sources/ — article/research/source notes
- people/ — people notes
- companies/ — company/tool/vendor notes
- archive/ — old or superseded material

Before creating a new page, search for an existing page first.
Do not store secrets, private keys, tokens, passwords, or raw credentials.
EOF

cat > index.md <<'EOF'
# Index

Start here for the brain.

Important areas:
- projects/
- repos/
- workflows/
- decisions/
- prompts/
- concepts/
- sources/
EOF

cat > schema.md <<'EOF'
# Schema

Preferred page shape:

# Title

## Current truth

The current best understanding.

## Details

Supporting details.

## Timeline / evidence

- YYYY-MM-DD — Source or event.

## Links

Related pages.
EOF

cat > log.md <<'EOF'
# Log

Append important brain maintenance notes here.
EOF

for d in inbox projects repos workflows decisions prompts concepts sources people companies archive; do
  cat > "$d/README.md" <<EOF
# ${d}

Purpose: see root RESOLVER.md.
EOF
done
```

Commit the initial brain repo:

```bash
git add .
git commit -m "Initialize AI brain"
```

Optional private GitHub remote later:

```bash
# Replace this with your actual private repo URL
git remote add origin git@github.com:YOUR_USER_OR_ORG/ai-brain.git
git branch -M main
git push -u origin main
```

## 8. Create a robust local MCP wrapper

This avoids PATH/cwd problems when Claude Code or Pi starts GBrain.

```bash
mkdir -p ~/bin

cat > ~/bin/gbrain-mcp <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

cd "$HOME/ai-brain"

export PATH="$HOME/.bun/bin:/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:$PATH"
export OLLAMA_HOST="${OLLAMA_HOST:-http://127.0.0.1:11434}"

exec gbrain serve
EOF

chmod +x ~/bin/gbrain-mcp
```

Test it:

```bash
~/bin/gbrain-mcp
```

Expected behavior: it will appear to hang silently. That is normal; stdio MCP servers wait for MCP JSON-RPC messages on stdin/stdout. Press `Ctrl+C`.

## 9. Connect GBrain to Claude Code locally

Use stdio MCP locally:

```bash
claude mcp add --scope user gbrain -- "$HOME/bin/gbrain-mcp"
```

Check:

```bash
claude mcp list
```

You want `gbrain` to show as connected.

Then test inside a repo:

```bash
cd ~/GitHub/YOUR_REPO
claude
```

Inside Claude Code:

```text
/mcp
```

Then ask:

```text
Use GBrain first. Search my brain for project context and summarize what you know before reading local files.
```

GBrain’s docs describe `gbrain serve` as the local stdio MCP path for Claude Code/Cursor/Windsurf-style clients, while HTTP mode is the remote/OAuth path. ([GitHub][1])

## 10. Add repo instructions so Claude Code actually uses it

In each project repo:

```bash
cd ~/GitHub/YOUR_REPO
touch CLAUDE.md
```

Append:

```bash
cat >> CLAUDE.md <<'EOF'

## GBrain usage

GBrain is available as an MCP server.

Before broad repo exploration, search GBrain first.

Use GBrain for:
- project architecture
- repo conventions
- prior decisions
- previous bugs/fixes
- implementation plans
- workflow notes
- related files, symbols, and subsystem context

Workflow:
1. Search GBrain for the feature, subsystem, symbol, or decision.
2. Use the result to narrow local file reads.
3. Use Grep/Glob/Read for exact code inspection after GBrain narrows the area.
4. If durable repo knowledge is learned, write it back to GBrain or update project docs.

Do not store secrets, credentials, private keys, tokens, customer secrets, or one-off noisy details in GBrain.
EOF
```

Commit:

```bash
git add CLAUDE.md
git commit -m "Add GBrain usage guidance"
```

Now start Claude Code:

```bash
claude
```

Use prompts like:

```text
Use GBrain first. Search for architecture decisions and repo conventions before reading files.
```

```text
Before implementing this, search GBrain for related prior decisions, bugs, and plans.
```

```text
We just learned something durable about this repo. Add a concise GBrain note for future sessions.
```

# Pi harness integration

Pi is a minimal terminal coding harness that is meant to be customized with extensions, skills, prompt templates, themes, and packages. Its site explicitly describes it as a minimal harness and says it can be adapted via packages/extensions. ([Pi.dev][6])

Pi itself may not expose MCP the same way Claude Code does out of the box. The clean path is to install a Pi MCP adapter package. The Pi package page for `pi-mcp-adapter` says it is an MCP adapter extension for Pi, installed with `pi install npm:pi-mcp-adapter`, and it reads standard MCP files like `.mcp.json` or `~/.config/mcp/mcp.json`. ([Pi.dev][7]) The adapter docs say it gives Pi a single proxy tool that discovers and calls MCP tools on demand rather than loading every MCP tool definition into the context. ([Pi MCP Adapter][8])

## 13. Install Pi on macOS

Use one of these. I’d use Bun since you already installed it:

```bash
bun add -g --ignore-scripts @earendil-works/pi-coding-agent
```

Or the Pi installer:

```bash
curl -fsSL https://pi.dev/install.sh | sh
```

Verify:

```bash
pi --version
```

---

## 14. Install MCP adapter for Pi

```bash
pi install npm:pi-mcp-adapter
```

Restart Pi after installing the package.

## 15. Create a standard MCP config for Pi

Use the wrapper script you already created.

```bash
mkdir -p ~/.config/mcp

cat > ~/.config/mcp/mcp.json <<EOF
{
  "mcpServers": {
    "gbrain": {
      "command": "$HOME/bin/gbrain-mcp",
      "args": []
    }
  }
}
EOF
```

Check the file:

```bash
cat ~/.config/mcp/mcp.json | jq
```

Start Pi in a project:

```bash
cd ~/GitHub/YOUR_REPO
pi
```

Inside Pi, try:

```text
/mcp
```

Then ask:

```text
Search the gbrain MCP server for project context and summarize what you find before reading local files.
```

If the adapter needs setup, run inside Pi:

```text
/mcp setup
```

The Pi MCP Adapter package page says it can scan/import host-specific configs and prefers `.mcp.json` or `~/.config/mcp/mcp.json`; it also notes `/mcp setup` as the setup flow when nothing is configured yet. ([Pi.dev][7])

```bash
cd ~/GitHub/YOUR_REPO
```

```bash
gbrain sync --help
```

Then try the current source/sync flow your installed version supports. Because GBrain’s code-indexing CLI has been moving quickly, I would not blindly depend on exact commands until `gbrain sources --help` and `gbrain sync --help` confirm them.

```text
GBrain = durable memory and decisions
```
